<h2><span class="NormalTextRun SCXW44149174 BCX8">What&#8217;s</span><span class="NormalTextRun SCXW44149174 BCX8"> changing?</span></h2>
<p><span>Previously, when inline images were added to a mail item’s signature or body using the </span><code>addFileAttachmentFromBase64Async</code>/<code>addFileAttachmentAsync</code> <span>method (</span><a href="https://learn.microsoft.com/javascript/api/outlook/office.messagecompose#outlook-office-messagecompose-addfileattachmentfrombase64async-member(1)"><span>Message Compose</span></a><span>) with </span><code>isInline</code><span> set to </span><strong>true</strong><span>, the images were uploaded to the server before the call completed. An EWS ID was then returned in the callback function for each image (for example, </span><code>AAkALgAAAAAAHYQDEapmEc2byACqAC/EWg0AW4xUByFbp0CroCNphMYEEAAGhqU3LwAAARIAEACS5VsVp3akQa7/BahqVbve</code><span>).</span><span> </span></p>
<p><span>Going forward, inline images will be uploaded to the server after the </span><code>addFileAttachmentFromBase64Async</code><span>/<code>addFileAttachmentAsync</code> call completes and the images are added to a signature or body. Because the </span><code>addFileAttachmentFromBase64Async</code><span>/<code>addFileAttachmentAsync</code></span><span><strong> </strong>call completes immediately without a network call, a temporary attachment ID is generated locally instead of an EWS ID. The attachment ID is prefixed with “addinId” (for example, </span><code>addinId:31c91015-920a-6fed-f663-970a0f9bc5cd</code><span>). The locally generated attachment ID is only temporary because an EWS ID is eventually generated for the attachment once it’s been uploaded to the server.</span><span> </span></p>
<p><span>The temporary attachment ID is fully supported with other attachment APIs, such as </span><code>getAttachmentsAsync</code><span> and </span><code>getAttachmentContentAsync</code><span>, for the duration of the current compose session (that is, while the mail item being composed is open). This behavior applies even after the attachments have been uploaded and assigned an EWS ID. Once the compose session ends, the temporary ID can no longer be used.</span><span> </span></p>
<p><span>Similar to the previous approach, users won&#8217;t be able to send the mail item until all inline attachments have been uploaded to the server. If a user attempts to send a message while attachments are still being uploaded, a dialog is shown informing them that the upload is still in progress.</span><span> </span></p>

<h2><span class="TextRun SCXW130371302 BCX8" lang="EN-IN" xml:lang="EN-IN"><span class="NormalTextRun SCXW130371302 BCX8">Where is it changing?</span></span><span class="EOP Selected SCXW130371302 BCX8"> </span></h2>
<p><span class="NormalTextRun SCXW43852924 BCX8">The updated approach improves </span><span class="NormalTextRun SCXW43852924 BCX8">signature scenarios</span><span class="NormalTextRun SCXW43852924 BCX8"> by making </span><span class="NormalTextRun SCXW43852924 BCX8">the addition of inline attachments</span><span class="NormalTextRun SCXW43852924 BCX8"> fast and resilient. It avoids waiting on network uploads before the signature can be applied and it ensures that a transient upload issue </span><span class="NormalTextRun SCXW43852924 BCX8">doesn’t</span><span class="NormalTextRun SCXW43852924 BCX8"> prevent the signature from appearing.</span></p>
<ul>
<li><strong><span class="TextRun SCXW223639731 BCX8" lang="EN-IN" xml:lang="EN-IN"><span class="NormalTextRun SCXW223639731 BCX8">Faster signature stamping (lower perceived latency)</span></span></strong><span class="TextRun SCXW223639731 BCX8" lang="EN-IN" xml:lang="EN-IN"><span class="NormalTextRun SCXW223639731 BCX8">. The signature can be set </span><span class="NormalTextRun SCXW223639731 BCX8">immediately</span><span class="NormalTextRun SCXW223639731 BCX8">, without waiting for each inline image to upload sequentially, so users see their signature sooner.</span></span><span class="EOP Selected SCXW223639731 BCX8"> </span></li>
<li><strong><span class="TextRun SCXW214030057 BCX8" lang="EN-IN" xml:lang="EN-IN"><span class="NormalTextRun SCXW214030057 BCX8">More reliable outcomes</span></span></strong><span class="TextRun SCXW214030057 BCX8" lang="EN-IN" xml:lang="EN-IN"><span class="NormalTextRun SCXW214030057 BCX8">. A failure to upload any one inline image no longer causes the entire signature stamping operation to fail, so users still get a signature and can continue composing and sending.</span></span><span class="EOP Selected SCXW214030057 BCX8"> </span></li>
</ul>
<h2><span class="TextRun SCXW46726944 BCX8" lang="EN-IN" xml:lang="EN-IN"><span class="NormalTextRun SCXW46726944 BCX8">Action </span><span class="NormalTextRun SCXW46726944 BCX8">required</span></span><span class="EOP Selected SCXW46726944 BCX8"> </span></h2>
<p><span>If your Outlook add-in relies on the EWS ID of an attachment to make Microsoft Graph calls, you must adjust your code to ensure that the </span><code>isServiceAccessible</code><span> property returns </span><span>true </span><span>before making the Graph calls.</span><span> </span></p>
<p><span>If your Outlook signature add-in calls </span><code>addFileAttachmentFromBase64Async</code>/<code><span>addFileAttachmentAsync</span></code><span>together with </span><a href="https://learn.microsoft.com/javascript/api/outlook/office.body#outlook-office-body-setsignatureasync-member(1)"><span>setSignatureAsync</span></a><span>, no changes are needed.</span><span> </span></p>
<h2><span>Timeline</span><span> </span></h2>
<p><span>Starting March 30, 2026, the changes to how inline attachments are handled will begin rolling out to production users. If needed, please make the necessary changes to your add-in before then.</span><span> </span></p>
